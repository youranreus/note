### NonFunctional Requirements

NFR1: 用户完成“打开产品到获得可分享便签链接”的核心流程应保持短路径和低等待感。
NFR2: 用户通过 `sid` 打开已有便签时，系统应在可接受时间内呈现内容。
NFR3: 便签保存与更新操作应在触发后快速反馈结果，避免用户误判为未成功。
NFR4: 分享主路径中的关键页面不应被非必要资源加载阻塞。
NFR5: 系统必须确保未授权用户不能修改便签内容。
NFR6: 编辑密钥不得以明文持久化方式存储。
NFR7: 登录用户身份与便签所有权关系必须被可靠识别和校验。
NFR8: 登录会话、编辑权限与收藏操作必须防止越权使用。
NFR9: 与登录和鉴权相关的数据传输必须通过安全通道完成。
NFR10: 固定 `sid` 链接必须具备稳定可访问性，不得频繁出现失效或异常不可达。
NFR11: 已保存内容必须具备可靠持久化，不应出现无提示丢失。
NFR12: 同一条便签的后续更新必须可靠反映为最新可见状态。
NFR13: 删除操作一旦确认完成，系统必须一致地反映不可恢复结果。
NFR14: 当出现失效链接、无权限或错误密钥等异常状态时，系统必须提供明确且可理解的反馈。
NFR15: 核心页面与关键操作应满足基础语义化要求。
NFR16: 用户应可通过键盘完成主要操作路径。
NFR17: 表单、按钮、弹窗与反馈信息应具备基础可感知性和可理解性。
NFR18: 系统必须能够稳定对接 SSO 登录与回调流程。
NFR19: 系统必须能够在前后端之间一致处理 `sid`、权限与收藏等核心业务状态。
NFR20: 本地便签与在线便签两种模式应保持明确边界，不得互相破坏数据语义。

### Additional Requirements

- 首个实现故事需要优先完成 `pnpm workspace` 基础骨架，包括 `apps/web`、`apps/api`、`packages/shared-types` 的仓库结构初始化。
- 前端必须采用 `Vue 3 + Tailwind CSS + alova.js + axios`，后端必须采用 `Fastify`，数据库为 `MySQL + Prisma`。
- 前端采用 SPA 架构，核心路由至少包括 `/`、`/auth/callback`、`/note/o/:sid`、`/note/l/:sid`，用户中心作为全局弹窗而非独立页面。
- API 必须采用 REST 风格，核心资源边界围绕 `notes`、`favorites`、`me/session` 与 `auth/callback` 组织。
- `notes.sid` 必须作为业务唯一键；内部关联统一使用内部主键，不能继续依赖自然键混合建模。
- 编辑密钥只能以安全摘要形式持久化，校验必须在服务端执行，前端不得保存明文密钥。
- 首次保存创建、按 `sid` 更新、用户首次登录建档、收藏关系写入都需要保证幂等或事务安全。
- 删除的产品语义是不可恢复终态；如果实现上使用软删除字段，也只能作为内部实现细节。
- 收藏关系需要复合主键或等价唯一约束，并支持级联删除，避免重复收藏和删除失败。
- 后端必须封装 `sso-facade`，统一处理 code exchange、userinfo、state、错误映射和会话建立，业务路由不能直接耦合三方 SSO 细节。
- 鉴权模型必须统一实现“资源所有权 + 编辑密钥”双轨授权，并对无权限、密钥错误、资源不存在、资源已删除、会话失效提供稳定错误语义。
- 前端状态分层必须明确：Pinia 只管理会话与 UI 状态，远端列表与详情数据由 alova 管理缓存、分页和失效重取。
- 在线便签与本地便签必须保持明确边界，本地模式仅存在于前端本地存储模块，不得与在线服务层混写。
- 工程基线至少需要 `lint`、`typecheck`、`test`、`build` 四类 CI 检查，并覆盖请求日志、SSO 异常、保存失败和权限异常的基本可观测性。
- 若历史数据存在重复 `sid`、明文密钥或旧外键链路，实施计划中必须包含数据库迁移与数据回填步骤。

### UX Design Requirements

UX-DR1: 建立基于 Tailwind 的最小设计 token 体系，至少统一颜色、排版、字号、圆角、边框、阴影、间距、状态色和动效时长。
UX-DR2: 实现轻量 foundation 组件层，至少包含输入框、主按钮、次按钮、弱按钮、表单容器、模态框、分段 tab、列表项、inline status/toast、empty state、loading state。
UX-DR3: 首页必须保持“单任务入口”结构，聚焦 `sid` 输入和“在线便签 / 本地便签”两个入口，避免把登录、收藏和管理入口前置到主流程之前。
UX-DR4: 当用户未输入 `sid` 时，界面必须自动生成并填充可用标识，避免空状态阻塞主路径。
UX-DR5: 在线便签结果页必须强化“固定入口对象”心智，清晰展示当前 `sid`、可分享状态、保存状态和编辑权限状态。
UX-DR6: 保存成功、复制链接成功、收藏成功、登录恢复成功等反馈应统一采用轻量且明确的反馈模式，优先 inline status 或轻量 toast。
UX-DR7: 口令错误、无权限、失效链接、资源已删除、SSO 回调失败等失败反馈必须具备明确原因和下一步建议，不能只展示泛化错误。
UX-DR8: 登录升级流程必须实现 `SsoConfirmModal` 和 `CallbackLoadingCard` 两类状态组件，分别承载登录确认和回跳加载恢复。
UX-DR9: 右上角状态入口必须实现 `AuthStatusPill`，统一未登录、已登录和打开个人中心三类状态，并具备清晰可读文本。
UX-DR10: 用户中心必须以 `UserCenterModal` + segmented tabs 承载“我的创建 / 我的收藏”，而不是独立后台页。
UX-DR11: “我的创建 / 我的收藏”列表项需要统一组件与空态文案；空列表时必须说明为空原因和建议下一步动作。
UX-DR12: 删除确认必须使用阻断式确认对话框，明确表达不可恢复后果；但普通保存、复制、收藏操作不应引入重确认。
UX-DR13: 所有 modal 打开时必须管理焦点、限制焦点陷阱并在关闭后返回触发源，tab 切换和主要动作必须支持键盘操作。
UX-DR14: 所有状态提示不能只依赖颜色，必须同时提供文字或图标表达；焦点态必须显式可见。
UX-DR15: 响应式设计采用移动优先策略，首页、在线便签页、SSO modal、回跳页和用户中心 modal 在移动端与桌面端都必须保持主路径完整。
UX-DR16: 页面与容器宽度应采用“最大宽度 + 百分比宽度”的响应式约束，不依赖固定像素宽度。
UX-DR17: 移动端高频点击区域，包括状态 pill、按钮、列表项等，最小可点击尺寸应接近或达到 `44x44px`。
UX-DR18: 组件开发时必须同时定义默认态、focus 态、error 态和 disabled 态，避免只实现默认外观。
UX-DR19: 按钮层级必须全局一致，同屏只能有一个唯一主任务 primary button，避免语义同级的多个 primary 动作并列。
UX-DR20: 页面级加载优先采用内容区域内状态，不使用全局大遮罩；回跳页使用居中 loading card，保证状态语义统一。

## Epic 1: 快速进入并分享便签

用户可以在极短路径内输入或自动生成 `sid`，进入在线或本地便签，持续更新内容，并通过稳定链接分享给他人查看最新版本。

### Story 1.1: 基于 starter template 建立可进入主流程的应用骨架

As a 访客用户,
I want 应用提供可用的首页、在线便签页和本地便签页骨架,
So that 我可以从同一个产品入口进入不同便签流程而不会被工程未完成状态阻塞。

**Acceptance Criteria:**

**Given** 项目仓库尚未完成重构骨架
**When** 开发者初始化产品实现
**Then** 前端应基于 `Vite Vue TypeScript starter` 初始化
**And** 后端应以 `Fastify TypeScript` 最小骨架完成 `apps/api` 初始化

**Given** starter template 初始化已经开始
**When** 仓库骨架被建立
**Then** 仓库中应存在 `apps/web`、`apps/api`、`packages/shared-types` 的基础结构
**And** 前端应提供 `/`、`/note/o/:sid`、`/note/l/:sid`、`/auth/callback` 的基础路由壳体

**Given** 用户第一次打开产品
**When** 首页完成加载
**Then** 页面应显示统一的应用壳体、右上状态入口和主内容容器
**And** 主内容容器在移动端和桌面端都应保持可用的响应式宽度约束

**Given** 组件基础层尚未建立
**When** 首页与便签页开始实现
**Then** 应存在最小 design token 与 foundation 组件基础
**And** 至少覆盖输入框、按钮、模态框、分段 tab、列表项、状态提示、空态、加载态和基础容器样式的一致性

**Given** foundation 组件被设计和实现
**When** 开发者定义组件交互状态
**Then** 每个高频组件至少应具备 default、focus、error 和 disabled 四类状态定义
**And** 这些状态不应留到页面开发完成后再补齐

### Story 1.2: 首页 SID 入口与模式选择

As a 访客用户,
I want 在首页输入或自动获得一个 `sid` 并选择在线或本地便签模式,
So that 我可以用最短路径进入我要使用的便签对象。

**Acceptance Criteria:**

**Given** 用户进入首页且输入框为空
**When** 页面初始化完成
**Then** 系统应自动准备一个可用的 `sid`
**And** 用户无需先手动输入 ID 也能继续主流程

**Given** 用户希望进入已有便签
**When** 在首页输入指定 `sid` 并选择在线便签
**Then** 系统应导航到对应的在线便签路由
**And** 该 `sid` 应作为后续读取与保存的唯一对象标识

**Given** 用户希望快速记录但不使用远端存储
**When** 在首页选择本地便签
**Then** 系统应导航到对应的本地便签路由
**And** 首页不应要求用户先登录或进入个人中心

**Given** 首页存在主要操作按钮
**When** 用户查看当前页面
**Then** 页面同屏只应存在一个当前主任务的 primary button 层级
**And** 输入区与两个入口按钮应构成清晰的单任务启动结构

### Story 1.3: 通过 SID 读取在线便签

As a 分享接收者,
I want 通过固定 `sid` 直接打开在线便签并查看最新内容,
So that 我无需登录或额外步骤就能消费分享结果。

**Acceptance Criteria:**

**Given** 某个 `sid` 已关联一条未删除的在线便签
**When** 访客直接访问 `/note/o/:sid`
**Then** 系统应按唯一 `sid` 读取该便签
**And** 内容页应直接展示该对象的最新已保存内容

**Given** 访客未登录
**When** 打开有效分享链接
**Then** 系统应允许正常阅读
**And** 不应因登录流程阻塞主阅读路径

**Given** 数据库和 API 被实现
**When** 读取便签详情
**Then** 后端应通过唯一 `sid` 而不是模糊查询返回单一对象
**And** 前后端应一致处理不存在、已删除和正常可读三类结果

### Story 1.4: 在线便签的首次保存与持续更新

As a 创建者,
I want 在同一 `sid` 下保存并持续更新在线便签内容,
So that 我分享出去的链接始终对应最新版本。

**Acceptance Criteria:**

**Given** 用户进入一个尚未保存过内容的在线便签 `sid`
**When** 首次执行保存
**Then** 系统应创建该 `sid` 对应的便签记录
**And** 之后再次进入同一链接时可以读取到刚保存的内容

**Given** 用户正在编辑一个已存在的在线便签
**When** 再次保存更新后的内容
**Then** 系统应在同一 `sid` 下更新最新内容
**And** 已分享链接不应发生变化

**Given** 前后端执行保存链路
**When** 创建或更新请求到达服务端
**Then** 服务端应保证按 `sid` 的幂等创建或更新语义
**And** 保存结果应以明确成功或失败反馈返回给前端

### Story 1.5: 可分享对象头部与主路径反馈

As a 创建者,
I want 在线便签页清楚展示当前对象状态和可分享链接,
So that 我能确认这个链接已经可用并放心发给别人。

**Acceptance Criteria:**

**Given** 用户位于在线便签页
**When** 页面展示对象级头部信息
**Then** 应清楚显示当前 `sid`、链接可分享状态、保存状态和编辑状态
**And** 这些状态应集中呈现在对象级头部附近而不是分散在多个角落

**Given** 用户完成保存、复制链接或其他主路径动作
**When** 系统反馈结果
**Then** 应优先使用 inline status 或轻量 toast
**And** 反馈既要简洁也要可被明确理解

**Given** 状态或操作反馈出现在页面中
**When** 用户通过颜色识别信息
**Then** 系统不能只依赖颜色表达结果
**And** 必须同时提供可读文字或等价语义提示

### Story 1.6: 本地便签独立模式

As a 访客用户,
I want 使用不依赖远端账号和数据库的本地便签模式,
So that 我可以在轻量场景下快速记录而不混淆在线分享对象。

**Acceptance Criteria:**

**Given** 用户从首页进入本地便签模式
**When** 在 `/note/l/:sid` 中编辑内容
**Then** 内容应保存在本地存储而不是远端在线便签数据源
**And** 本地模式与在线模式的数据读取和保存逻辑必须明确分离

**Given** 用户再次打开相同的本地便签 `sid`
**When** 本地存储中存在对应内容
**Then** 页面应恢复本地最新内容
**And** 不应误读或覆盖远端在线便签数据

**Given** 本地便签页面提供主操作
**When** 用户在移动端或桌面端使用该页面
**Then** 核心内容区与主按钮应保持完整可用
**And** 页面布局不应因屏宽变化改变核心流程语义

## Epic 2: 登录升级与受控编辑

用户可以通过 SSO 建立身份，在登录创建者默认编辑权与编辑密钥共享编辑权之间完成受控编辑，并在登录后恢复自己的身份与上下文。

### Story 2.1: SSO 登录入口与回跳恢复

As a 需要升级能力的用户,
I want 在需要登录时进入 SSO 并在回跳后恢复当前上下文,
So that 我可以无缝获得账户能力而不必重新开始主流程。

**Acceptance Criteria:**

**Given** 未登录用户点击右上状态入口或触发需要登录的操作
**When** 系统需要用户建立身份
**Then** 应显示 `SsoConfirmModal`
**And** 该弹窗只表达必要的登录升级信息而不打断用户理解当前上下文

**Given** 用户确认进行 SSO 登录
**When** 浏览器进入 `/auth/callback`
**Then** 页面应展示 `CallbackLoadingCard` 处理回跳中间态
**And** 在成功建立会话后返回来源页或合理默认页

**Given** modal 或回跳页被打开
**When** 用户通过键盘操作或辅助技术使用页面
**Then** 焦点管理、关闭行为和状态文本都必须可感知
**And** 登录流程中的状态提示不能只依赖视觉装饰

### Story 2.2: 已登录创建者的默认编辑权

As a 已登录创建者,
I want 以我的账户身份创建和编辑属于我的在线便签,
So that 我可以默认受控地维护自己分享出去的内容。

**Acceptance Criteria:**

**Given** 用户已建立有效登录会话
**When** 首次保存一个新的在线便签
**Then** 系统应把当前用户识别为该便签的创建者
**And** 后续基于同一身份访问该便签时默认具有编辑权

**Given** 已登录创建者打开自己创建的便签
**When** 执行编辑和保存
**Then** 系统应允许修改并成功持久化内容
**And** 前后端都应基于资源所有权做一致校验

**Given** 用户会话失效或身份不匹配
**When** 尝试以创建者身份更新便签
**Then** 系统应拒绝该修改请求
**And** 返回清楚、稳定的权限错误语义

### Story 2.3: 编辑密钥的设置与共享编辑

As a 创建者或未登录发起者,
I want 为便签设置编辑密钥并通过密钥授予共享编辑能力,
So that 我可以在不开放公开编辑的前提下让他人协作更新内容。

**Acceptance Criteria:**

**Given** 创建者希望把编辑权共享给他人
**When** 为某条便签设置编辑密钥
**Then** 服务端应只保存密钥的安全摘要
**And** 前端不得在持久化存储中保存明文密钥

**Given** 未登录用户通过编辑密钥模式创建便签
**When** 首次保存该便签
**Then** 系统应允许在无账户身份前提下建立带密钥的编辑对象
**And** 必须清楚表达遗失编辑密钥后不可恢复编辑权

**Given** 协作者持有正确编辑密钥
**When** 进入对应便签并提交更新
**Then** 系统应允许该修改
**And** 错误密钥与无密钥两种失败状态必须可区分反馈

### Story 2.4: 查看权与编辑权的统一授权状态

As a 便签访问者,
I want 系统明确区分我当前是只读、创建者可编辑还是密钥可编辑,
So that 我能理解自己能做什么而不会误操作。

**Acceptance Criteria:**

**Given** 用户进入在线便签页
**When** 系统完成对象和权限状态判断
**Then** 页面应明确展示当前是可查看、可编辑还是需要密钥
**And** 该表达应与对象级头部状态保持一致

**Given** 用户没有编辑权限
**When** 尝试触发编辑或保存
**Then** 系统应拒绝实际写入
**And** UI 应给出可行动的下一步提示而不是模糊失败

**Given** 前后端都参与权限判断
**When** 请求被处理
**Then** 查看权限与编辑权限的规则必须一致
**And** 不允许前端显示可编辑但服务端实际拒绝且无解释的状态错位

## Epic 3: 收藏与个人资产管理

已登录用户可以收藏他人便签，在“我的创建 / 我的收藏”中统一回访和进入目标内容，把零散链接沉淀为个人资产。

### Story 3.1: 登录门槛下的收藏动作

As a 分享接收者,
I want 在登录后收藏重要便签,
So that 我可以把值得回访的内容沉淀到自己的账户资产中。

**Acceptance Criteria:**

**Given** 用户正在查看一条他人分享的在线便签
**When** 该用户已登录并点击收藏
**Then** 系统应创建该用户与便签之间的收藏关系
**And** 收藏结果应立即反馈到当前页面状态中

**Given** 用户尚未登录
**When** 在便签页尝试收藏
**Then** 系统应要求先完成登录
**And** 登录成功后应能回到原上下文继续该动作

**Given** 用户重复收藏同一条便签
**When** 系统处理收藏请求
**Then** 收藏关系应保持幂等
**And** 不应产生重复记录或给用户造成异常错误

### Story 3.2: 用户中心弹窗与我的创建

As a 已登录用户,
I want 通过个人中心查看我创建过的便签列表,
So that 我可以从账户视角重新进入自己维护的内容对象。

**Acceptance Criteria:**

**Given** 用户已登录
**When** 点击右上 `AuthStatusPill`
**Then** 系统应打开 `UserCenterModal`
**And** 默认以清晰、轻量的弹层语义承载个人中心而不是跳转后台页

**Given** 个人中心被打开
**When** 用户进入“我的创建”tab
**Then** 系统应按合理排序展示自己创建的便签列表
**And** 列表项应统一展示可识别信息并可直接进入目标便签

**Given** “我的创建”列表为空
**When** 页面渲染空状态
**Then** 系统应说明为空原因
**And** 应给出创建第一条便签的下一步建议

### Story 3.3: 我的收藏列表与上下文返回

As a 已登录用户,
I want 在个人中心查看我的收藏并返回目标便签,
So that 我可以从收藏资产中快速回到需要继续阅读的内容。

**Acceptance Criteria:**

**Given** 用户已登录且存在收藏记录
**When** 切换到“我的收藏”tab
**Then** 系统应展示该用户的收藏列表
**And** 收藏列表的排序、样式和点击区域应保持一致的列表组件规范

**Given** 用户在个人中心点击某条收藏记录
**When** 系统执行跳转
**Then** 应关闭 `UserCenterModal` 并进入对应便签页面
**And** 用户应感知为从资产层回到对象层，而不是进入新的后台流程

**Given** “我的收藏”列表为空
**When** 页面展示空状态
**Then** 系统应说明当前没有收藏内容
**And** 应提示用户可在阅读便签时执行收藏

### Story 3.4: 个人中心的可访问性与状态一致性

As a 已登录用户,
I want 个人中心的 tab、列表和关闭行为在键盘与触控场景都清楚可用,
So that 我可以稳定地管理资产而不会因交互细节丢失上下文。

**Acceptance Criteria:**

**Given** 用户通过键盘操作个人中心
**When** 在 tab、列表项和关闭按钮之间移动
**Then** 焦点顺序应符合视觉与操作顺序
**And** tab 切换、关闭弹层和选择列表项都必须支持键盘完成

**Given** 用户通过触控或鼠标操作个人中心
**When** 点击状态 pill、列表项或关键操作按钮
**Then** 高频点击区域尺寸应接近或达到 `44x44px`
**And** 不应出现视觉可见但实际难以点击的热区

**Given** 用户关闭个人中心
**When** modal 被关闭
**Then** 焦点应返回到触发该弹层的 `AuthStatusPill`
**And** 用户不应失去之前所在页面上下文

**Given** 个人中心处于移动端或桌面端
**When** 页面根据屏宽调整布局
**Then** modal 尺寸和列表布局可以增强变化
**And** “我的创建 / 我的收藏”两类资产结构不能因响应式而改变语义

## Epic 4: 生命周期管理与可信反馈

用户可以安全地结束便签生命周期，明确理解删除不可恢复的后果，并在失效链接、权限失败、错误密钥等边界场景中获得清楚、可信的反馈。

### Story 4.1: 删除确认与受权删除

As a 有管理权限的用户,
I want 在清楚理解后果的前提下删除便签,
So that 我可以明确结束一条内容的生命周期。

**Acceptance Criteria:**

**Given** 用户对某条便签拥有删除权限
**When** 触发删除动作
**Then** 系统必须先展示阻断式确认对话框
**And** 对话框应明确说明删除不可恢复的后果

**Given** 用户在确认对话框中最终确认删除
**When** 系统执行删除请求
**Then** 便签应被标记为不可再访问的终态
**And** 删除结果应向用户返回明确成功或失败反馈

**Given** 用户没有相应管理权限
**When** 尝试删除便签
**Then** 系统应拒绝该请求
**And** 反馈应明确说明是权限问题而不是泛化错误

### Story 4.2: 删除后的失效链接与不可恢复语义

As a 分享接收者或创建者,
I want 已删除便签在原链接上表现为明确失效,
So that 我不会误以为内容只是暂时加载失败或仍可恢复。

**Acceptance Criteria:**

**Given** 某条便签已经被成功删除
**When** 任意用户再次访问原 `sid` 链接
**Then** 系统应返回已删除或已失效的明确状态
**And** 不应继续展示原内容

**Given** 创建者尝试找回已删除便签
**When** 通过原有分享链接或个人入口访问
**Then** 系统不得提供恢复内容的路径
**And** 产品语义上必须保持删除不可恢复

**Given** 实现层采用软删除字段
**When** 系统处理已删除记录
**Then** 对外表现仍必须等价于不可恢复终态
**And** 前后端错误语义不能把软删除暴露成可回收功能

### Story 4.3: 边界异常的统一反馈体系

As a 任意使用者,
I want 在失效链接、错误密钥、无权限和登录回跳失败时得到清楚反馈,
So that 我能判断当前状态并知道下一步该做什么。

**Acceptance Criteria:**

**Given** 用户遇到不存在便签、已删除便签、错误密钥、无编辑权限或 SSO 回跳失败
**When** 系统返回异常结果
**Then** 页面应显示可理解的原因说明
**And** 每种异常至少应提供一个明确的下一步建议

**Given** 异常反馈出现在页面中
**When** 用户通过桌面端、移动端或辅助技术查看
**Then** 信息必须具有可读文本
**And** 不得只通过颜色、位置或短暂动画表达错误

**Given** 保存、删除、登录和权限校验都可能失败
**When** 产品处理这些边界场景
**Then** 反馈样式和语气应保持一致
**And** 不应让用户在不同页面看到彼此冲突的状态语义

**Given** 页面需要展示加载中的中间状态
**When** 用户处于普通页面内容加载或回跳处理过程中
**Then** 非回跳场景应优先使用内容区域内加载状态
**And** 不应默认使用全局大遮罩覆盖整个应用
