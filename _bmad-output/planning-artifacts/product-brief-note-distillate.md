# Product Brief Distillate: Note

## Product Intent

- Note 的核心定位是“双态便签产品”：同一产品同时承接“在线便签”和“本地便签”两类记录意图。
- 核心价值主张不是高级编辑能力，而是“低门槛开始记录”以及“登录后把记录沉淀为个人资产”。
- 目标用户是有高频碎片记录需求的个人用户，而不是团队协作场景。

## Problem Signals

- 现有笔记/文档工具常把登录、创建文档、选择空间放在记录动作之前，打断“先记一下”的自然心流。
- 纯本地记录工具虽然快，但跨设备访问、收藏沉淀和长期回访能力弱。
- 用户的真实问题是“快速落笔”和“之后找回”同时成立，单一模式通常只能解决一半。

## Experience Model

- 首页是低决策入口，设计稿显示只有 ID 输入、在线便签、本地便签和登录状态区。
- 用户可输入 ID 直达，也可在为空时由前端生成随机 ID，这体现了“降低进入成本”的产品原则。
- 登录不是前置门槛，而是查看个人面板、收藏、管理创建内容时才显性的升级动作。
- 登录后有“我的创建 / 我的收藏”双 tab，说明产品希望把零散内容转化为可管理资产。
- SSO 回调页单独存在，意味着登录链路需要被产品化地处理，而不是简单跳转后静默完成。

## MVP Scope Signals

- MVP 应至少包含：首页、在线便签详情、本地便签详情、SSO 回调页、用户信息面板。
- MVP 必备能力包括：便签按 SID 稳定访问、在线便签 CRUD、基础口令校验、收藏/取消收藏、我的创建分页、我的收藏分页。
- 用户信息面板是全局弹窗而非独立路由，这意味着“管理资产”应保持轻量，不打断主流程。
- 本地便签独立路由 `/note/l/:sid`，在线便签路由 `/note/o/:sid`，体现模式区分要保留。

## Technical Context

- 前端固定选型为 `Vue3 + TailwindCSS + alova.js + axios`。
- 后端固定选型为 `Fastify`，SSO 集成为 `@reus-able/sso-utils`。
- 数据库为 MySQL，访问层计划使用 Prisma。
- alova 被定位为请求状态与缓存层，Pinia 仅承接会话信息和 UI 状态，不应重复托管大列表数据。

## Data/Architecture Constraints

- `notes.sid` 必须全局唯一，这是产品层“单 ID 访问单便签”成立的基础。
- 现状中 `sid` 无唯一约束，读使用 `findFirst`、删使用 `deleteMany`，是当前产品风险之一。
- `key` 明文存储被视为不可接受，重构目标是只存 `key_hash`。
- 用户关系要从外部自然键 `sso_id` 迁移为内部 `users.id` 关联，以支撑更稳健的扩展。
- 收藏关系应使用 `(note_id, user_id)` 复合主键，天然保证幂等。
- 删除策略存在待定点：`deleted_at` 已在目标模型中预留，但是否启用软删除需要产品确认。

## User/Business Success Hints

- 需要重点验证首页到首条便签创建的转化效率，这比传统 DAU 更能反映“低门槛入口”是否成立。
- 需要跟踪在线入口和本地入口的使用分布，判断双态是否真的各自承接不同需求，而不是造成认知噪音。
- 登录后访问“我的创建”与“我的收藏”的比例，是“资产承接”价值是否被感知的关键指标。
- 需要观察回访行为，如再次打开同一 SID、收藏后回看、从首页进入历史内容，来判断产品是否超越一次性工具。

## Differentiation Notes

- 差异化来自体验顺序，不来自功能数量：先记录，后管理；先使用，再登录。
- Note 不是重型笔记软件，也不是仅有本地存储的临时输入框，而是试图覆盖“临时记录到轻量沉淀”的中间地带。
- 若要对外表达竞争力，应强调“几乎零摩擦启动”和“登录后内容资产化”这两个连续价值，而不是泛泛谈便签功能。

## Risks and Open Questions

- `@reus-able/sso-utils` 的实际接入方式尚需 PoC 验证，可能影响登录链路稳定性和交付节奏。
- 历史数据库若存在重复 `sid`，上线前必须有清洗策略，否则产品承诺的唯一访问会失效。
- 本地便签是否长期保持独立产品语义，还是未来与在线便签进一步融合，目前仍是产品决策点。
- 便签删除是否采用软删除、是否需要找回能力，目前没有最终产品结论。
- MVP 暂未定义分享、协作、搜索等能力边界的后续优先级，PRD 阶段需要明确。

## Rejected or Deferred Directions

- 首版不应扩展到复杂协作、团队空间、重型富文本编辑或通知体系，否则会稀释核心验证目标。
- 不建议在 MVP 里把登录设为前置要求，这会直接破坏产品最核心的低摩擦价值。
- 不建议过早引入大而全的知识管理叙事；当前更适合定位为高频轻量记录入口。
